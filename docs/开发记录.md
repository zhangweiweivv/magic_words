## 📅 Phase 2.5 测试功能（2026-03-29）

### 功能需求
主人提出：每天复习完成后，需要增加测试环节，测试通过才算当日任务完成。

### 测试规则
| 项目 | 设定 |
|------|------|
| **测试形式** | 混合模式：选择题 + 拼写题 |
| **题目数量** | 10题 = 5道选择题 + 5道拼写题 |
| **选择题** | 显示中文意思，选正确的英文单词（4选1） |
| **拼写题** | 显示中文意思，输入英文单词 |
| **选项来源** | 从今日复习的10个单词中抽取 |
| **大小写** | 忽略（booking 和 Booking 都算对） |
| **通过标准** | 错题重做，做到全对为止 |
| **触发时机** | 用户对当日所有单词都点"记住了"后，自动进入测试 |

### 测试流程
```
复习阶段                    测试阶段
┌────────────────────┐     ┌────────────────────┐
│ 翻卡 → 记住了/没记住 │ ──▶ │ 10题混合测试        │
│ （10个单词）        │     │ 错题循环重做        │
└────────────────────┘     └────────────────────┘
                                   │
                                   ▼
                           ┌────────────────────┐
                           │ 🎉 测试通过！       │
                           │ 今日任务完成        │
                           └────────────────────┘
```

### 新增组件
- `client/src/components/QuizMode.vue` — 测试模式组件

### 技术实现
1. 复习完成时，保存已复习单词列表 `reviewedWords`
2. 进入测试模式，生成10道题目（5选择 + 5拼写，随机打乱）
3. 选择题：从10个单词中随机抽4个作为选项
4. 答错的题目加入 `wrongQueue`，当前轮次结束后重做
5. 全部答对后显示完成界面

## 📅 Phase 2.6 艾宾浩斯升级（2026-03-29）

### 需求变更
主人提出：
1. 原3次复习改为4次（0天→1天→3天→7天）
2. 测试配比随复习阶段动态调整
3. Day 7 测试通过后自动移到「已记住」

### 艾宾浩斯流程（最终版）

| 阶段 | 间隔 | 时间点 | 测试配比 |
|------|------|--------|----------|
| Day 0 | 首次学习 | 当天 | 拼写3 + 选择7 |
| Day 1 | +1天 | 第2天 | 拼写5 + 选择5 |
| Day 4 | +3天 | 第5天 | 拼写7 + 选择3 |
| Day 11 | +7天 | 第12天 | 纯拼写10题 |

**设计理念**：越往后拼写占比越高，强化输出记忆

### 完整学习周期（以10个单词为例）

```
Day 0:  学习10个单词 → 翻卡确认 → 测试(3拼写+7选择) → 通过 ✓
        ↓ 间隔1天
Day 1:  复习10个单词 → 翻卡确认 → 测试(5拼写+5选择) → 通过 ✓
        ↓ 间隔3天
Day 4:  复习10个单词 → 翻卡确认 → 测试(7拼写+3选择) → 通过 ✓
        ↓ 间隔7天
Day 11: 复习10个单词 → 翻卡确认 → 测试(纯拼写10题) → 通过 ✓
        ↓
        🏆 完全掌握！自动移到「已记住单词.md」
```

## 📅 Phase 3.6 UI 问题修复 + SVG 鱼群（2026-03-29 21:00-21:26）

### 修复问题

**1. 导航栏顶部深色区域**
- **现象**：滚动到顶部时，"可可的单词魔法屋"上方有深色区域
- **根因**：`MusicPlayer.vue` 内部 `.music-player { position: relative }` 覆盖了 `App.vue` 的 `.global-music-player { position: fixed }`，导致音乐播放器占据页面流空间，把导航栏推下去
- **修复**：`.global-music-player { position: fixed !important }`

**2. 鱼群动画不显示**
- **现象**：只看到气泡，看不到鱼群
- **根因**：Vue scoped CSS 会给 `@keyframes` 加后缀（如 `swimAcross-fa3aadae`），但 JS 内联设置 `animationName: 'swimAcross'` 不会加后缀，导致动画名不匹配
- **修复**：改用 CSS 类（`.anim-straight`、`.anim-wobble`）来应用动画

**3. 鱼游动方向倒置**
- **现象**：鱼看起来是"倒着游"的
- **根因**：emoji 鱼默认头朝左，动画的 scaleX 逻辑搞反了
- **修复**：分离两个 CSS 变量
  - `--direction`：控制移动方向（1=往右，-1=往左）
  - `--flip`：控制翻转（-1=翻转让鱼头朝右，1=不翻转）

**4. 色调不协调**
- **现象**："每日探险任务"和"我的宝藏"按钮是橙色系，与海洋蓝不搭
- **修复**：
  - 任务卡片：`coral+starfish` → `ocean-medium+seaweed`（蓝绿渐变）
  - 按钮：`coral` → `secondary`（浅蓝边框）
  - 统计卡片边框：统一改为蓝/绿色调

### 新功能：SVG 鱼群

**需求**：主人反馈 emoji 鱼太卡通，要求改成真实风格的 SVG

**技术细节**：
- 所有鱼使用 SVG 渐变填充
- 带眼睛高光和身体阴影

| 文件 | 改动 |
|------|------|
| `App.vue` | `position: fixed !important` 修复音乐播放器定位 |

| `SwimmingFish.vue` | 完全重写为 SVG 鱼群，6种普通鱼 + 4种大型生物 |

### 学到的教训

1. **Vue scoped CSS + @keyframes + JS 内联样式 = 坑**
   - 在 scoped 组件中，动画名要通过 CSS 类来应用
   - 不要用 JS 内联设置 `animationName`

2. **组件内部样式可能覆盖外部类选择器**
   - 即使不是 scoped，后加载的样式优先级相同时会覆盖
   - 用 `!important` 或更具体的选择器确保覆盖

3. **CSS 动画会覆盖内联 transform**
   - 把翻转逻辑移到 CSS 变量，在 @keyframes 中统一使用

## 📅 守护进程修复（2026-04-01）

### 问题现象
网站打不开，服务无法访问。

### 排查过程
1. `launchctl list | grep keke` → 服务在跑（PID 34331，状态 0）
2. `lsof -i :3000 -i :3001 -i :3003` → 端口无监听
3. 查 stdout 日志 → 发现错误：
   - `Error: listen EADDRINUSE: address already in use 0.0.0.0:3001`
   - `Error: Port 5173 is already in use`

### 根因分析
LaunchAgent plist 直接跑 `npm run dev`，通过 `concurrently` 启动多个子进程（node server + vite）。

**问题**：macOS 杀 LaunchAgent 时只杀顶层 npm 进程，`concurrently` 启动的子进程变成**孤儿进程**继续占着端口。新进程启动时端口被占，绑定失败。

### 修复方案

新增守护进程专用启动脚本 `scripts/start-daemon.sh`：

**三重保护机制：**
1. **启动前清理** — 杀掉所有残留的 keke-vocab 进程，逐端口检查并强制释放
2. **trap 信号处理** — 收到 SIGTERM/SIGINT/SIGHUP 时杀掉整个进程组（`kill -- -$$`），不留孤儿
3. **端口强制释放** — 如果杀进程后端口仍被占（3001/5173），`kill -9` 强制清理

**plist 改动：**
- 原来：`/opt/homebrew/bin/npm run dev`
- 现在：`/bin/bash scripts/start-daemon.sh`

### 修改文件

| 文件 | 改动 |
|------|------|
| `scripts/start-daemon.sh` | **新增** — 守护进程启动脚本，含清理+trap+端口检查 |
| `~/Library/LaunchAgents/com.keke.vocab.plist` | ProgramArguments 改为调用 start-daemon.sh |

## 📅 艾宾浩斯批次调度重构（2026-04-01）

主人发现原来的逻辑有两个问题：
1. 翻卡点“记住了”就推进进度，但可能没真正记住也会点
2. 不同阶段的单词会混在一起复习

### 改动一：测试通过后才推进进度

**之前**：翻卡点“记住了” → 直接推进艾宾浩斯进度

**现在**：
- 翻卡点“记住了” → 只加积分，不推进进度
- 测试通过后 → 统一推进该批所有单词的进度

**理念**：翻卡是“自我声称”，测试才是“真正验证”

### 改动二：批次调度，不混合不同阶段

**之前**：取到期的 + 新词，凑15个混在一起

**现在**：
- 一天只做一批，不混合
- 有到期批次 → 做该批次
- 没有到期 → 取15个新词

**优先级**：毕业考(Day3) > 复习1(Day1) > 新词(Day0)

### 完整时间线示例

```

Day 1: 批次A 复习1 → 下次 Day 4
Day 2: 空闲 → 批次B 首学 → 下次 Day 3
Day 3: 批次B 复习1 → 下次 Day 6
Day 4: 批次A 毕业考 ✅ 结业！
Day 5: 空闲 → 批次C 首学
Day 6: 批次B毕业考 + 批次C复习1 碰撞 → 毕业考优先
...
```

**规律**：每5天完成一个批次的完整周期，偶尔碰撞时延后一天

## 2026-04-04 更新

### 1. 音节拆分算法重写

**问题**：原有的纯元音模式拆分不准确
- `enclosed → en·clo·sed` ❌（应为 `en·closed`）
- `regarded → re·ga·ded` ❌（应为 `re·gard·ed`）
- `coming → co·ming` ❌（应为 `com·ing`）
- `whole → who·le` ❌（单音节词，不应拆）

**修复方案**：
1. **后缀识别层**：先识别常见后缀（-ed, -ing, -tion, -ment, -ness, -ful, -less, -ure, -age, -ive, -ence 等 20+），拆出 stem + suffix 分别处理
2. **-ed 特殊处理**：只有在 t/d 后才是独立音节（regarded → re·gard·ed），其他情况合并到前一音节（enclosed → en·closed）
3. **-ing 双写辅音**：running → run·ning，coming → com·ing
4. **Silent-e 改进**：whole 正确识别为单音节；-le 后缀只在辅音+le 时保留（ta·ble, lit·tle）
5. **开/闭音节平衡规则**：
   - 词首 CV 音节 → 开音节（re·gard, pa·per, ti·ger）
   - 非词首 CVC 音节 → 闭音节（in·ter, ac·tiv）
   - 尾音节过短时保持开音节（com·pa·ny）
6. **短语跳过**：包含空格的短语不拆分音节
