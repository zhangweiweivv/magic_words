## 1. 背景与目标

主人希望将现有的「可可的单词魔法屋」扩展为“多模块学习宇宙”，并新增古诗文背诵/理解模块。

### 1.1 核心目标

1) **新增 Portal 首页（可可魔法屋）**：作为“地图/城堡入口”，可进入多个子站点/子模块。
2) **新增 古诗文模块**：数据源来自 Obsidian 的「小红本」目录（公用数据源，只读不写），从“寅集”开始逐集攻克。
3) **艾宾浩斯进度引擎**：古诗文与单词不同——每篇文章独立维护自己的艾宾浩斯周期（阶段数+间隔）与详细日志。
4) **规范工程化**：有 design、有 plan、有全面 code review、可审计。

### 1.2 非目标（当前阶段不做）

- 不在 Phase 1 强制做“背诵正确性自动判定/语音识别打分”。（初期信任用户自评标记“今日已学完”）
- 不将古诗文学习进度写回 `小红本/`（该目录为公共只读数据源，未来糖糖也可能复用）。

## 3. 数据源与数据目录规划

### 3.1 公用只读数据源：小红本

- 路径：`/Users/vvhome/vv_obsidian/vv_obsidian/小红本/`
- 约束：**只读，不写入学习进度**。
- 当前：`小红本/寅集.md`（目录索引）
- 未来：继续新增 `卯集.md`、`辰集.md` 等。

#### 3.1.1 集目录索引（示例：寅集.md）

- `寅集.md` 目前为目录：分题材（先秦诸子/古文/诗词）+ 篇名 + 页码。
- 设计要求：后续每个集的索引文件结构允许有 3/4/5... 种题材。

#### 3.1.2 每篇文章的内容文件

为了让网站可以“自给自足”学习（原文/注释/译文/赏析/拼音/TTS注音），每篇文章需要独立 md 文件。

#### 3.1.3 不同“集”之间的学习顺序（跨集串行）

设计要求：**整体背诵/攻克流程在不同集之间完全串行**。

- 只在“当前集（active collection）”内进行新学推荐与推进；
- 当前集全部篇目 `graduated` 后，才进入下一集；
- 跨集顺序：按 `小红本/` 目录下不同集的顺序即可。

实现建议（确定性、可扩展）：

1) 发现集：扫描 `小红本/` 根目录下的 `*.md`（排除子目录），每个 `XX集.md` 视为一个 collection。
2) 集排序：
   - 若文件名匹配地支（子丑寅卯辰巳午未申酉戌亥），则按地支自然顺序排序；
   - 否则回退为文件名的 localeCompare/字典序排序。
3) 当前集选择：选择排序后第一个“尚未全部毕业”的集作为 active collection。

> 说明：串行只影响“新学推荐/推进集切换”，不改变单篇文章内部的艾宾浩斯推进与碰撞规则。

推荐结构：

- `小红本/寅集/01-论语八章.md`
- `小红本/寅集/02-老子三章.md`
- ...

每篇 md 文件建议格式：

```md
# 桃花源记
> 作者：陶渊明 | 朝代：东晋 | 体裁：古文

## 原文
...

## 拼音（可选显示）
...

## 注释
- ...

## 译文
...

## 赏析
...

## TTS注音（用于朗读，解决多音字/停顿）
...

## 元数据
- 字数：320
- 题材：古文
```

> 注：拼音与 TTS 注音的目标不同：拼音服务“阅读辅助”，TTS 注音服务“机器朗读准确性与停顿节奏”。

## 5. “题材轮询”的新学推荐逻辑

### 5.0 前置：轮询的作用域

- 题材轮询仅在 **当前 active collection（当前集）** 内生效。
- 若 active collection 的所有文章均毕业，则切换到下一集；题材轮询游标在新集内重新初始化。

### 5.1 规则

- 当前集有 N 种题材，则按 **1:1:1...（N份）轮询**。
- 若某题材已无可新学文章（已全部进入 in_progress 或 graduated），跳过该题材继续轮询。

### 5.2 状态持久化

需要在 `config.json` 或独立状态文件记录：

- `rotationCursor`：上一次推荐到哪个题材
- 每题材的“下一篇候选列表/游标”（可按目录顺序，或按难度/字数排序，默认按目录顺序）

## 6. 艾宾浩斯：每篇独立周期 + 自适应默认推荐

### 6.1 核心差异

- 单词：可批次共享一个状态
- 古诗文：**每篇文章独立维护**
  - `totalStages`（阶段数）
  - `intervals`（阶段间隔序列）
  - `currentStage`、`nextDueDate`

### 6.2 自适应难度计算（默认值）

难度取 1-4，综合两类因子（长度、文体）：

- 长度分：≤50字=1，51-150=2，151-300=3，>300=4
- 文体分：诗词=1，古文（唐宋）=2，古文（先秦）=3，先秦诸子=4
- 难度 = round(长度分*0.5 + 文体分*0.5)

### 6.3 默认推荐表

- ⭐ 简单：`stages=3`，`intervals=[1,2,5]`
- ⭐⭐ 中等：`stages=3`，`intervals=[1,3,7]`
- ⭐⭐⭐ 较难：`stages=4`，`intervals=[1,3,7,14]`
- ⭐⭐⭐⭐ 困难：`stages=5`，`intervals=[2,4,7,14,21]`

## 7. 碰撞（多篇到期）与逾期策略

### 7.1 现实约束

- 每天晨读最多 20 分钟
- 每周只有 3-4 天做语文，其他天读英语
- 因此“到期碰撞”是常态，需要合理的“未完成顺延”机制

### 7.2 碰撞处理原则

1) **今天到期的篇目全部列出来**（包括逾期）。
2) 用户自主选择今天做几篇。
3) 完成者：推进阶段，下一次到期日基于“实际完成日”计算。
4) 未完成者：保持当前阶段不变，标记逾期（overdueDays 增加）。

### 7.3 下一次到期日计算

- `nextDueDate = actualCompletionDate + intervals[nextStage]`
- 避免“基于原计划日”导致后续间隔被压缩引发连锁碰撞。

### 7.4 优先级排序

默认排序（从高到低）：

1) 逾期文章（overdueDays 降序）
2) 最后一轮/快毕业阶段（stage 越靠后越优先）
3) 正常到期（stage 降序）
4) 同条件下短篇优先（字数升序）

## 8. 家长动态调整周期：严谨的状态一致性

### 8.1 规则总览

- **历史不可变**：已完成阶段的 `scheduledDate/actualDate` 不回溯修改。
- 调整只影响未来：重算 `nextDueDate`、未来阶段配置。

### 8.2 调整间隔（变长/变短）

- 仅对“当前待执行阶段/未来阶段”生效
- 立即重算 `nextDueDate`
- 若 `today > nextDueDate`，则该篇变为“逾期”，出现在今日列表

### 8.3 调整阶段数（变多/变少）

- 增加阶段：延长周期，继续推进
- 减少阶段：
  - 若 `currentStage >= newTotalStages`：直接毕业（写入 `config_changed` + `graduated`）
  - 否则正常继续（可能使当前待执行阶段成为毕业考）

### 8.4 已毕业文章

- 已毕业文章默认锁定配置（禁止直接增加阶段导致“反毕业”）。
- 如需重新学：提供显式的 `reset` 功能（清空该篇进度并记录日志）。

## 9. 状态与日志：可审计、可追溯

### 9.1 主状态（每篇文章）

每篇文章状态建议包含：

- 基本信息：collection、title、题材、字数、difficulty
- 周期信息：stages、intervals、currentStage、nextDueDate
- 历史信息：stageHistory（计划 vs 实际 vs 逾期）
- 统计：overdueCount

### 9.2 事件日志（append-only）

- 采用 JSONL（每行一个 JSON）便于追加与排查：`events-log.jsonl`
- 事件类型：
  - `first_learn` / `stage_completed` / `graduated`
  - `deferred` / `overdue_resolved`
  - `config_changed` / `reset`

### 9.3 UI 可视化

- 每篇文章详情页展示完整时间线与事件列表
- 首页展示：今日到期/逾期数量与排序依据（尽可能解释化）
